## Question

After granting the exact skeleton and literal payload, can the remaining
`208` copy-source fields be generated by decoder-visible source rules or
by source-context selectors that generalize?

## Ledger

- Copy events: `208`.
- Oracle target-chunk matching sources: `597` total.
- Matching sources min/median/max: `1` / `2` / `14`.
- Single-source events: `78`.
- Multi-source events: `130`.
- Canonical source is earliest matching in `200/208` events.
- Canonical source is latest matching in `85/208` events.
- Oracle rank bits among matching sources: `232.902`.
- Raw absolute source bits: `2550.594`.

The strong earliest-matching signal is not a generator by itself: it
requires knowing which target chunk the copy must reproduce.

## Policy Gate

- Decoder-visible policies tested: `7`.
- Target-aware oracle controls tested: `3`.
- Best decoder-visible policy: `previous_end_else_earliest`.
- Best decoder-visible chunk hits: `8/208`.
- Best decoder-visible sequential exact books: `1/60`.
- Best oracle control chunk hits: `208/208`.

The oracle controls prove the candidate set can reproduce the books when
the target chunk is already known. They do not remove source choice from
the generation recipe.

## Context Gate

- Context families tested: `11`.
- Best context: `book_mod10_x_length`.
- Best full-fit chunk hits: `173/208`.
- Best full-fit source-exact hits: `173/208`.
- Best net vs raw absolute source bits: `199.919` bits.
- Prefix/holdout cover-all cells: `0/5`.

The best full-fit context is still source-bearing and does not cover all
copies. Prefix/holdout does not produce a cover-all source selector.

## Decision

- No copy-source generator is promoted.
- The `208` source fields remain external after granting skeleton and literal payload.
- Target-aware earliest/matching-source controls are parser/oracle controls, not generation rules.
- This does not change the compression bound.
- Row0 remains exogenous and unchanged.
- No plaintext, translation, semantic reading, or case reopening is introduced.
